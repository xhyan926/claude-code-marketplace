# go-sdk-planner

## 技能用途

本技能用于将 OBS SDK Go 开发任务分解为可独立完成的子任务。适用于：

- 新功能开发（如添加新的 API 方法）
- Bug 修复（如内存泄漏、逻辑错误）
- 代码重构（如提取公共逻辑、优化性能）

## 前置条件

1. **项目环境正确**：工作目录为 `/Users/xhyan/project/SDKS/huaweicloud-sdk-go-obs`
2. **需求明确**：用户提供了清晰的功能需求、Bug 描述或重构目标
3. **上下文充分**：已阅读相关代码文件，理解现有架构和约定

## 输入格式

用户提供以下信息之一：

### 功能开发
```
"为 ObsClient 添加批量删除对象功能"
```

### Bug 修复
```
"修复 PutObject 在大文件上传时的内存泄漏"
```

### 代码重构
```
"重构 auth.go 中的签名逻辑"
```

## 输出格式

### 1. 主任务文件：`SUBTASKS.md`
```markdown
# 任务名称

## 任务描述
详细说明任务目标和范围

## 子任务列表
1. 子任务 1 - 设计阶段
2. 子任务 2 - 实现阶段
3. 子任务 3 - 测试阶段

## 总体进度
[ ] 子任务 1
[ ] 子任务 2
[ ] 子任务 3
```

### 2. 子任务目录：`subtasks/task-01/`
每个子任务包含：
- `TASK.md` - 子任务详细描述
- `STATUS` - 状态标记（pending）
- `IMPLEMENTATION.md` - 实施计划
- `TEST_PLAN.md` - 测试计划
- `API_CHANGE_TRACKER.md` - API 变更跟踪

## 任务分解原则

### 粒度控制
- 每个子任务应在 **1-3 天** 内完成
- 避免过度拆分（<1 天）或过度聚合（>3 天）
- 考虑开发、测试、验证的完整流程

### 依赖管理
- 子任务之间应保持最小依赖
- 必须依赖时，在 TASK.md 中明确说明
- 使用阻塞标记确保执行顺序

### 幂等性保证
- 检查文件是否存在，避免重复创建
- 通过 STATUS 文件标记执行状态

## 执行流程

### 第一步：需求分析
1. 与用户确认任务需求
2. 识别关键依赖和约束
3. 确定任务类型（新功能/Bug/重构）

### 第二步：任务分解
1. 根据任务类型应用相应的分解策略
2. 创建 SUBTASKS.md
3. 为每个子任务创建目录和必需文件

### 第三步：验证
1. 验证子任务数量合理（4-6 个）
2. 确认每个子任务的粒度合适
3. 检查依赖关系清晰

## 注意事项

### 项目约定参考
根据 `CLAUDE.md`，遵循以下约定：

1. **测试命名规范**：使用 BDD 风格
   ```
   Should_return_error_When_invalid_input_Given_empty_string
   ```

2. **架构层次**：
   - 客户端方法：`client_*.go`
   - 特性层：`trait_*.go`
   - HTTP 层：`http.go`, `auth*.go`
   - 模型层：`model_*.go`

### 常见陷阱

1. **过度设计**：避免为简单任务创建过多子任务
2. **状态不一致**：确保 STATUS 文件与实际状态同步
3. **测试缺失**：每个子任务都必须有对应的测试

## 质量检查清单

每个任务分解完成后，检查：

- [ ] SUBTASKS.md 已创建
- [ ] 子任务数量在 4-6 之间
- [ ] 每个子任务包含所有必需文件
- [ ] 依赖关系明确
- [ ] 时间估算合理（1-3 天）
