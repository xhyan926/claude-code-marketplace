# code-reviewer

## 技能用途

本技能用于对 Go 代码进行全面审查，确保代码质量。适用于：

- 单元测试完成后的代码审查
- 子任务完成前的质量检查
- 多轮代码迭代中的持续审查

## 审查专家知识

本技能在以下领域具有专家级知识：

1. **Go 语言最佳实践**（Effective Go）
2. **对象存储领域知识**（OBS API、对象操作）
3. **网络编程**（错误处理、超时、重试、并发）
4. **Clean Code 原则**（命名、函数长度、复杂度）
5. **设计模式**（Go 中的设计模式应用）
6. **并发安全性**（race condition、锁使用、goroutine 管理）

## 前置条件

1. **代码已完成**：待审查的代码已实现
2. **测试已通过**：单元测试和集成测试已通过
3. **STATUS 文件**：子任务状态为 completed 或即将标记为 completed

## 审查检查点

### 1. Go 语言最佳实践（Effective Go）

- [ ] 命名规范
  - 包名小写，无下划线
  - 导出函数/变量首字母大写
  - 非导出函数/变量首字母小写
  - 接口命名以 er 结尾

- [ ] 错误处理
  - 错误总是最后一个返回值
  - 错误立即检查，不忽略
  - 错误信息清晰、有上下文
  - 使用 errors.Wrap/WithMessage 包装错误

- [ ] 并发安全
  - 共享数据使用 sync.Mutex 或 sync.RWMutex 保护
  - 避免数据竞争
  - 正确使用 channel
  - 合理使用 context 取消操作

- [ ] 接口设计
  - 接口小而专注
  - 接口在调用方定义，不在实现方
  - 避免接口污染

### 2. 对象存储领域知识

- [ ] API 使用正确
  - 使用正确的 OBS API 方法
  - 参数设置正确
  - 返回值处理恰当

- [ ] 对象操作规范
  - Bucket/Object 命名规范
  - 元数据处理正确
  - ACL/权限设置正确

- [ ] 分块上传
  - 分块大小合理
  - 并发上传控制
  - 断点续传支持

### 3. 网络编程

- [ ] 超时处理
  - HTTP 请求设置超时
  - context 使用正确
  - 超时后资源清理

- [ ] 重试机制
  - 可重试错误正确识别
  - 指数退避实现
  - 最大重试次数限制

- [ ] 连接管理
  - 连接池使用正确
  - 连接关闭处理
  - 资源泄漏检查

### 4. Clean Code 原则

- [ ] 函数设计
  - 单一职责
  - 函数长度合理（<50 行）
  - 参数数量合理（<5 个）

- [ ] 命名清晰
  - 变量名有意义
  - 避免缩写
  - 布尔变量名清晰（is/has/can）

- [ ] 代码复杂度
  - 圈复杂度合理（<10）
  - 嵌套层级浅（<4 层）
  - 避免深层嵌套

### 5. 设计模式

- [ ] 模式应用恰当
  - 单例模式（如适用）
  - 工厂模式（如适用）
  - 策略模式（如适用）
  - 选项模式（Options Pattern）

- [ ] 避免过度设计
  - 不滥用设计模式
  - 保持简单
  - YAGNI（You Aren't Gonna Need It）

### 6. 并发安全性

- [ ] 数据竞争检查
  - 通过 `go test -race` 无数据竞争
  - 共享数据保护
  - 原子操作正确使用

- [ ] 锁使用
  - 锁范围最小
  - 避免死锁
  - 读写锁选择正确

## 审查流程

### 第 1 轮：代码检视

1. 读取待审查代码
2. 按照检查点逐一检查
3. 记录发现的问题
4. 生成 REVIEW_COMMENTS.md

### 第 2 轮：开发修复

1. 开发者根据 REVIEW_COMMENTS.md 修复问题
2. 更新代码
3. 更新 REVIEW_COMMENTS.md 的修复状态

### 第 3 轮：再次检视

1. 检查修复是否完整
2. 验证是否引入新问题
3. 更新 REVIEW_COMMENTS.md

### 重复直到达成一致

- 所有问题已修复或解释
- 审查者和开发者达成一致

## 问题状态

| 状态 | 说明 |
|------|------|
| pending | 待修复 |
| fixed | 已修复 |
| waived | 已豁免 |
| explained | 已解释 |

## 审查通过条件

满足以下条件时，审查通过：

1. 无严重问题
2. 一般问题全部修复或解释清楚
3. 代码符合 Go 最佳实践
4. 无明显的安全隐患

## 注意事项

### 审查态度

- 建设性地提出问题
- 提供具体的改进建议
- 认可代码亮点和好的实践
- 尊重开发者的设计决策

### 沟通方式

- 问题清晰描述，不模糊
- 提供代码示例说明
- 欢迎讨论和反馈
- 对事不对人

### 时间安排

- 第 1 轮审查：1-2 小时
- 开发者修复：按问题数量估计
- 后续审查轮次：30-60 分钟
